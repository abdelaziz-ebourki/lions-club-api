# Implementation Plan: JWT Principal Wrapper

**Branch**: `010-jwt-principal-wrapper` | **Date**: 2026-07-16 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/010-jwt-principal-wrapper/spec.md`

## Summary

Replace the raw String principal in `JwtAuthenticationFilter` with a typed `UserPrincipal` record containing userId, email, role, firstName, lastName. This enables:
- Direct injection via `@AuthenticationPrincipal UserPrincipal principal` in controllers
- Method-level security expressions referencing `principal.userId`, `principal.role`, etc.
- Elimination of boilerplate `getCurrentUser()` methods that query UserRepository

Key changes:
1. Create `UserPrincipal` record in `com.lionsclub.api.security`
2. Update `JwtTokenProvider` to include email, firstName, lastName in JWT claims
3. Update `JwtAuthenticationFilter` to construct `UserPrincipal` from claims
4. Refactor `EventController` and `RsvpController` to use `@AuthenticationPrincipal`
5. Update AuthController to use typed principal
6. Update integration tests to expect `UserPrincipal` instead of String

No database changes required (no Flyway migration). No API contract changes (OpenAPI unchanged).

## Phase 0: Research (`research.md`)

### Research Tasks

| Task | Question | Status |
|------|----------|--------|
| R1 | Does `UserPrincipal` need to implement `Principal` interface or is a plain record sufficient for `@AuthenticationPrincipal`? | ✅ Resolved |
| R2 | How to add custom claims (email, firstName, lastName) to JWT via Auth0 java-jwt library? | ✅ Resolved |
| R3 | Spring Security `@EnableMethodSecurity` configuration for `@AuthenticationPrincipal` resolution? | ✅ Resolved |
| R4 | Test patterns for mocking `UserPrincipal` in controller integration tests? | ✅ Resolved |

### Research Findings (to be filled in `research.md`)

**R1**: Plain record is sufficient. Spring Security's `AuthenticationPrincipalArgumentResolver` resolves any principal type from `Authentication.getPrincipal()`. No interface implementation required.

**R2**: Auth0 java-jwt: `JWT.create().withClaim("email", email).withClaim("firstName", firstName).withClaim("lastName", lastName).withClaim("role", role.name())...`. Extract via `decoded.getClaim("email").asString()`.

**R3**: `@EnableMethodSecurity` already present in `SecurityConfig`. `@AuthenticationPrincipal` works out of the box with any principal type. SpEL expressions like `principal.userId` work if principal has getter/userId() method.

**R4**: Use `@WithMockUser` for basic auth or custom `@WithMockUserPrincipal` annotation. For integration tests, create test `UserPrincipal` and set in `SecurityContextHolder`.

## Phase 1: Design

### Data Model (`data-model.md`)

**UserPrincipal Record**

```java
package com.lionsclub.api.security;

import java.util.UUID;
import com.lionsclub.api.domain.user.Role;

public record UserPrincipal(
    UUID userId,
    String email,
    Role role,
    String firstName,
    String lastName
) implements java.security.Principal {
    @Override
    public String getName() {
        return email; // or userId.toString()
    }
}
```

**JWT Claims Structure** (updated)

| Claim | Type | Source | Required |
|-------|------|--------|----------|
| sub | String (UUID) | User.id | Yes |
| role | String | User.role | Yes |
| email | String | User.email | Yes |
| firstName | String | User.firstName | Yes |
| lastName | String | User.lastName | Yes |
| iat/exp | NumericDate | JwtConfig | Yes |
